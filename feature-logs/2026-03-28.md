## 1. Real Workspace Index

### Overview

Implemented a real backend workspace index so the explorer no longer rebuilds itself by recursively walking the filesystem on each refresh. The new index tracks files, folders, ignore rules, metadata, timestamps, and hashes, then serves tree/search consumers from cached state.

| `src-tauri/src/lib.rs` | Registered workspace index commands and initialized persistence at app startup |

### Dependencies Added

None.

### Implementation Details

#### Workspace Index Architecture

The index keeps a single active workspace in memory for fast runtime access and uses the existing app database for persistence:

```rust
static WORKSPACE_INDEX: OnceLock<Mutex<Option<WorkspaceIndex>>> = OnceLock::new();
static WORKSPACE_INDEX_DB_PATH: OnceLock<PathBuf> = OnceLock::new();
```

Each indexed entry stores:
- absolute normalized path
- relative path / parent relationship
- file-or-directory flag
- size
- modified timestamp
- content hash for files

#### Incremental Update Flow

```
┌──────────────┐     ┌────────────────────┐     ┌──────────────────────┐
│ OS file      │────►│ notify watcher     │────►│ workspace_index      │
│ change       │     │ debounce task      │     │ apply_file_changes() │
└──────────────┘     └────────────────────┘     └──────────┬───────────┘
                                                           │
                                                           ▼
                                                ┌──────────────────────┐
                                                │ explorer refresh uses│
                                                │ indexed tree data    │
                                                └──────────────────────┘
```

### Architecture Decisions

1. **Single active in-memory workspace** - Matches the current app model and avoids premature multi-root complexity.
2. **Index-first explorer refreshes** - Eliminates repeated directory walks from normal UI refresh paths.
3. **Watcher-driven updates** - Keeps the cache warm without forcing full rebuilds during normal editing.

## 2. Persistent Workspace Cache

### Overview

Extended the workspace index so it persists into the same SQLite database already used by chat/session storage. Reopening a project can now hydrate the index from disk instead of requiring a full cold rebuild every time.

### Files Modified

| File | Changes |
|------|---------|
| `src-tauri/src/commands/workspace_index.rs` | Added SQLite tables for roots and entries, load/save logic, cache summary, clear-cache command, and persistence toggle |
| `src-tauri/src/lib.rs` | Initialized the workspace index with the existing `chat_state.sqlite` path during startup |

### Implementation Details

#### SQLite Schema

Two tables were added inside the existing app DB:

```sql
CREATE TABLE IF NOT EXISTS workspace_index_roots (
    root_path TEXT PRIMARY KEY,
    ignore_rules_json TEXT NOT NULL,
    last_indexed_at INTEGER NOT NULL
);

CREATE TABLE IF NOT EXISTS workspace_index_entries (
    root_path TEXT NOT NULL,
    rel_path TEXT NOT NULL,
    path TEXT NOT NULL,
    name TEXT NOT NULL,
    is_dir INTEGER NOT NULL,
    parent_rel_path TEXT,
    size INTEGER NOT NULL,
    modified_ms INTEGER NOT NULL,
    hash TEXT,
    PRIMARY KEY (root_path, rel_path)
);
```

#### Persistence Behavior

- When enabled, the index loads from SQLite first.
- If no cache exists, it rebuilds from disk and saves the result.
- Watcher-driven updates resave the index after changes.
- Clearing the cache removes both DB rows and the current in-memory index.

### Architecture Decisions

1. **Reuse existing SQLite DB** - Avoided introducing a second local storage system.
2. **Load-on-demand** - Persistence improves reopen performance without complicating startup with background workers.
3. **Toggleable persistence** - Supports low-RAM / low-value projects without removing the in-session index.

## 4. Ripgrep Search Engine

### Overview

Replaced the old Rust-side file-content scan with a ripgrep-backed search engine aimed at VS Code-style performance on large repos and monorepos. The existing search UI was preserved.

### Files Modified

| File | Changes |
|------|---------|
| `src-tauri/src/commands/search_commands.rs` | Replaced file-by-file scan logic with `rg --json` execution, result parsing, truncation handling, and fallback search path |

### Implementation Details

#### Search Pipeline

```
┌──────────────┐     ┌──────────────┐     ┌────────────────────────┐
│ Search UI    │────►│ Tauri command │────►│ ripgrep --json         │
└──────────────┘     └──────────────┘     └──────────┬─────────────┘
                                                     │
                                                     ▼
                                      ┌────────────────────────────┐
                                      │ matched files re-read only │
                                      │ to build byte ranges and   │
                                      │ context lines for the UI   │
                                      └────────────────────────────┘
```

#### Behavior

- uses `rg` for candidate/match discovery
- respects include/exclude globs
- enforces file-size caps
- stops when the global result cap is reached
- falls back to indexed Rust scanning if `rg` is unavailable
- returns usable file paths for open/replace flows

### Architecture Decisions

1. **Use ripgrep as the engine** - The fastest path to monorepo-scale search without writing a custom index/searcher.
2. **Keep UI payload shape stable** - Allowed the frontend to stay mostly unchanged.
3. **Hybrid result assembly** - `rg` finds matches quickly; Rust enriches only matched files with context/byte ranges.

## 5. Explorer and Drag/Reveal Fixes

### Overview

While working through workspace/search improvements, two user-facing explorer bugs were fixed: Windows reveal-in-explorer resolving to `This PC`, and dragging items from a folder back to the project root being swallowed by file-row drop handlers.

### Files Modified

| File | Changes |
|------|---------|
| `src-tauri/src/commands/file_commands.rs` | Canonicalized Windows paths and adjusted Explorer invocation for reveal behavior |
| `src/components/file-tree/FileItem.tsx` | Limited drag/drop handlers to directory nodes so root drops can bubble correctly |

### Architecture Decisions

1. **Canonicalize before Windows Explorer calls** - Explorer is sensitive to path formatting and argument shape.
2. **Directories-only drop interception** - File rows should not consume drop events intended for the root zone.
