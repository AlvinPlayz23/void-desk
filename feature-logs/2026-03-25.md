### Implementation Details

#### OAuth Flow

The new auth module follows the browser-based ChatGPT login flow:

```text
VoiDesk settings
   │
   ▼
codex_start_login()
   │
   ▼
System browser -> OpenAI auth
   │
   ▼
localhost callback
   │
   ▼
token exchange + auth.json persistence
```

The backend stores:

- `access_token`
- `refresh_token`
- `expires_at_ms`
- `chatgpt_account_id`

This keeps Codex credentials out of frontend persistence.

#### Provider Routing

The backend now branches on `providerType` instead of assuming every provider is OpenAI-compatible:

```rust
match provider_type.as_deref() {
    Some("codex_subscription") => { /* Codex provider */ }
    _ => { /* OpenAI-compatible provider */ }
}
```

This affects chat, inline completions, connection testing, and debug commands.

### Architecture Decisions

1. **Separate provider type** - Codex was implemented as its own provider because the auth model, request format, and SSE event shapes differ materially from OpenAI-compatible chat completions.
2. **Rust-side token storage** - ChatGPT OAuth tokens are stored in a local backend-managed auth file instead of Zustand/localStorage.
3. **Shared auth, per-preset provider selection** - OAuth login is shared at the app level, while provider choice remains selectable in both universal mode and preset mode.

## 2. Codex Protocol Compatibility Fixes

### Overview

After the initial provider landed, two protocol bugs showed up immediately in real usage. Multi-turn conversations failed because previous assistant turns were replayed with the wrong content type, and tool-using turns could go blank because the provider only looked at `response.completed` instead of consuming `response.output_item.done` during streaming.

### Root Cause Analysis

1. **Multi-turn 400 error** - Codex rejected assistant history items serialized as `input_text`; previous assistant messages must be replayed as `output_text`.
2. **Blank tool-first turns** - The provider ignored `response.output_item.done`, but Codex emits function calls there before the final `response.completed`.
3. **Server-side 503s observed** - `503 no healthy upstream` was confirmed as an upstream availability error. Retry handling for open-request failures was briefly tested and then removed by user request, so no automatic open-request retry remains in the final implementation.

### Files Modified

| File | Changes |
|------|---------|
| `src-tauri/src/sdk/provider/codex_subscription.rs` | Fixed assistant history serialization, added SSE item parsing, added completion fallback parsing for `output_item.done`, then removed experimental open-request retry handling |

### Implementation Details

#### Assistant History Serialization

Previous assistant turns are now replayed as `output_text`:

```rust
let assistant_message =
    message_to_codex_input_with_text_kind(&message, "output_text");
```

This fixed the `Invalid value: 'input_text'` error on follow-up prompts.

#### SSE Parsing

The provider now consumes Codex-native SSE events directly:

```text
response.output_text.delta
response.output_item.done
response.completed
```

Tool calls are emitted from `response.output_item.done`, and text deltas are streamed from `response.output_text.delta`. `response.completed` is now treated as the final usage/done boundary plus fallback output source.

#### Completion Fallback

For non-streaming-style response assembly, the provider reconstructs the final assistant result from streamed SSE items if the final `response.completed.output` is empty.

### Architecture Decisions

1. **Use output-item events as authoritative for tools** - This matches observed Codex behavior and avoids blank tool rounds.
2. **Keep final `response.completed` as fallback only** - Useful for usage accounting and as a safety net when the output array is present.
3. **Do not retain open-request retry wrapper** - A retry loop for Codex open-request failures was removed to avoid masking upstream/server-side behavior after user feedback.

### Problems Encountered

| Problem | Solution |
|---------|----------|
| Follow-up prompts failed with `input_text` validation error | Replayed assistant history as `output_text` |
| Tool-first prompts produced blank output | Parsed `response.output_item.done` directly for function calls |
| Codex `response.completed.output` could be empty even when tools were emitted | Reconstructed results from earlier SSE items |
| Upstream `503 no healthy upstream` failures | Confirmed as server-side; no open-request retry retained in final implementation |

## 3. Provider-Specific Model Defaults

### Overview

The settings UI originally kept the existing OpenAI-compatible model array even after switching the provider type to ChatGPT OAuth. This made Codex mode show the wrong model list until the user edited models manually. The settings layer now swaps to Codex defaults when provider type changes.

### Files Modified

| File | Changes |
|------|---------|
| `src/stores/settingsStore.ts` | Added Codex default model list and provider-aware model replacement logic |
| `src/components/settings/SettingsPage.tsx` | Updated provider-switch handlers in both single-provider and preset mode to swap to provider-specific defaults |

### Implementation Details

The Codex defaults are now:

```typescript
const DEFAULT_CODEX_MODELS: AIModelConfig[] = [
    { id: "gpt-5.4", name: "gpt-5.4", supportsImages: false },
    { id: "gpt-5.4-mini", name: "gpt-5.4-mini", supportsImages: false },
    { id: "gpt-5.3-codex", name: "gpt-5.3-codex", supportsImages: false },
    { id: "gpt-5.1-codex-mini", name: "gpt-5.1-codex-mini", supportsImages: false },
];
```

Both universal mode and preset mode use provider-aware switch logic when the provider type changes.

### Architecture Decisions

1. **Provider defaults live in the store** - Keeps model defaults centralized and reusable between migration, UI, and preset creation.
2. **Switch behavior is explicit** - When the provider type changes, the visible model list is replaced with the new provider’s defaults instead of trying to infer compatibility from the old list.
