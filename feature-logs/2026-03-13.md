## 1. Image Attachment Timeout Fix

### Overview

Image attachments to AI chat messages were consistently timing out (45s stream open timeout, then 90s completion timeout) despite the image payloads being properly optimized and transmitted. The root cause was the provider/streaming path hanging on multimodal requests. The fix involved ensuring the non-streaming fallback path was correctly triggered and adding comprehensive debug logging to trace the request lifecycle.

### Root Cause Analysis

The debugging process revealed a multi-layered issue:

1. **Initial symptoms**: `Provider stream did not open within 45s` - requests hung waiting for SSE stream response
2. **First attempt (non-streaming fallback)**: Added `messages_include_inline_images()` to detect images and use `client.complete()` instead of `client.stream()` for multimodal requests
3. **Fallback triggered but still timed out**: The `Iteration 1` log showed `"building non-streaming multimodal request"` confirming the fallback was working, but the 90s timeout still fired
4. **Final fix**: The actual issue was insufficient logging to diagnose what happened after the HTTP request was sent. Added transport-level logging and increased timeout to 180s.

### Implementation Details

#### Multimodal Detection

The key function that enables the fallback:

```rust
fn messages_include_inline_images(messages: &[Message]) -> bool {
    messages.iter().any(|message| {
        match message.content.as_ref() {
            Some(MessageContent::Multipart(parts)) => {
                parts.iter().any(|part| matches!(part, MessagePart::Image { .. }))
            }
            _ => false,
        }
    })
}
```

#### Agent Request Building

In `run_streaming_with_handle`, the request type is determined by image presence:

```rust
let contains_inline_images = messages_include_inline_images(&messages);
let request = agent.build_request(messages.clone(), !contains_inline_images);
// ^ stream=false when images present, stream=true when text-only

if contains_inline_images {
    // Use non-streaming completion with 180s timeout
    let response = timeout(
        Duration::from_secs(MULTIMODAL_COMPLETION_TIMEOUT_SECONDS),
        agent.client.complete(request),
    ).await;
} else {
    // Use SSE streaming with 45s timeout
    let stream = timeout(
        Duration::from_secs(STREAM_OPEN_TIMEOUT_SECONDS),
        agent.client.stream_with_debug(request, debug_raw),
    ).await;
}
```

#### Multipart Message Construction

`Message::user_multipart()` builds OpenAI-compatible multipart content:

```rust
pub fn user_multipart(text: String, image_attachments: Vec<InlineImageAttachment>) -> Self {
    let mut parts = Vec::new();
    if !text.is_empty() {
        parts.push(MessagePart::Text { text });
    }
    for attachment in image_attachments {
        parts.push(MessagePart::Image {
            image_url: ImageUrl {
                url: attachment.data_url,  // data:image/png;base64,...
                detail: attachment.detail,
            },
        });
    }
    Self {
        role: "user".to_string(),
        content: Some(MessageContent::Multipart(parts)),
        ...
    }
}
```

### Architecture Decisions

1. **Non-streaming for multimodal** - Some providers (particularly OpenRouter/Moonshot) hang when trying to stream multimodal requests. Using standard completion avoids this issue entirely.

2. **Increased timeout (180s)** - Large images (even after optimization to ~77KB data URLs) can take longer to process. 180s gives sufficient buffer.

3. **Kept debug logging** - Retained all transport and provider-level logging for future troubleshooting. These logs are essential for diagnosing provider-specific issues.

4. **Transport-level visibility** - Adding logs at `post_text()` and `parse_non_stream_response()` level ensures we can see exactly when/where HTTP requests succeed or fail.

### Problems Encountered

| Problem | Solution |
|---------|----------|
